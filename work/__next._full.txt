## What it is

A SOC assistant I built on my own time. The idea was simple: stop digging through documentation every time an alert fires and just ask a question in plain English.

The core is a RAG pipeline — Pinecone as the vector store, LangChain managing the retrieval and chains, Gemini 1.5 Pro doing the reasoning. It ingests security documentation, playbooks, and telemetry context, and surfaces relevant answers in seconds instead of minutes of manual searching.

It's not a product. It's a personal project that I run against real data and actually use.

## Why I built it

I work in security operations. A lot of the slow parts of alert triage aren't complex thinking — they're retrieval. Finding the right playbook, checking what a particular indicator means in this environment, confirming whether a behavior is expected for a given user. That's lookup work, and lookup work is exactly what RAG is good at.

I wanted to see how much of that I could automate without the output becoming something you can't trust. That second part turned out to be the hard part.

## Stack

- **Orchestration**: LangChain — manages retrieval chains and the agentic loop
- **Model**: Google Gemini 1.5 Pro — large context window was the deciding factor over GPT-4
- **Vector store**: Pinecone — low-latency retrieval at scale
- **Integrations**: Microsoft Sentinel, Defender XDR, Azure Logic Apps

## What actually worked

The retrieval speed was genuinely good from the start. Once the embeddings were in Pinecone, lookup latency was fast enough that the bottleneck shifted entirely to the model response time, which is what you want.

The natural language interface worked better than I expected for well-defined questions — "what's the expected login behavior for this user?", "show me playbook steps for lateral movement detection" — that kind of thing.

## What didn't

Hallucination in forensic context is a real problem. Early versions would confidently answer questions that weren't well-represented in the retrieval results. I spent a significant amount of time tuning retrieval quality — chunk sizing, embedding strategy, score thresholds — before the answers were reliable enough to act on.

I also initially tried to make it too autonomous. Fully automated playbook execution sounds useful until a false positive triggers an account lockdown at 2am. The current version flags high-confidence actions for one-click confirmation rather than running them silently.

## Results (personal use)

Triage time on familiar alert types dropped significantly — the lookup work that was eating 15-30 minutes now takes seconds. For novel or complex incidents I still do full manual investigation, which is the right call. The assistant is a first-pass filter, not a replacement for thinking.

## What this is

A passion project built because hosting friends for coffee had become genuinely chaotic. Multiple orders, specific customizations, no memory of who asked for oat milk — it was more stressful than it should be.

So I built an order management system. Guests scan a QR code, see the menu, customize their order, and submit it. I get a kitchen display showing the queue with all the details. It sounds like overkill, and it is, but it works perfectly.

## How it works

**Guest side** — a mobile-first menu where guests choose their drink, add customizations (milk type, shots, temperature), and place the order. Loads instantly, no app install needed, works as a PWA if they want it on their home screen.

**Host side (KDS)** — a kitchen display that shows the current queue. Orders appear immediately when placed, I can mark them as "preparing" or "ready", and the guest's screen updates in real time without a refresh.

Firebase Firestore real-time snapshots handle the live updates — when an order status changes on the KDS, it propagates to the guest view in under a second.

## What I actually learned from this

Firebase Firestore's real-time subscriptions are genuinely good for this use case. The tricky part was handling the edge cases — what happens when the host closes the KDS while orders are in flight, stale order states, offline behavior. None of it was hard, but it required thinking through the state machine more carefully than the happy path suggests.

Also: the first version had a bug where two guests placing orders within milliseconds of each other could get the same order number. Fixed it with a Firestore transaction. Classic concurrency problem, embarrassingly obvious in retrospect.

## Stack

- Next.js 15 (App Router)
- Bun
- Firebase Firestore (real-time)
- PWA

## What this is

My personal Claude Code configuration that I've been refining since I started using it heavily for both security automation and portfolio work. The default Claude Code setup is fine. This is what happens after you use it seriously for a few months and start fixing the parts that annoy you.

## The problem with default Claude Code

Two things break down quickly:

1. **No memory across sessions.** Every new conversation starts cold. You re-explain your stack, your preferences, your project context. It adds up.

2. **No workflow enforcement.** Claude will happily skip tests, write unsafe code, or use npm instead of bun if you don't have rules that say otherwise.

## What I built

**Session context system** — Four files per project (`context-goals.md`, `context-decisions.md`, `context-progress.md`, `context-gotchas.md`) that persist across sessions. A PreCompact hook assembles them before context gets trimmed. A SessionStart hook injects the summary at the start of every new session. Claude picks up where it left off.

**Custom hooks** — Shell commands that fire on tool use events:
- PostToolUse runs Prettier and tsc after every file edit
- PreToolUse blocks dev servers from running outside a tmux session
- EvaluateSession extracts patterns at session end
- SuggestCompact warns when approaching context limits

**Agent definitions** — Specialized agents for recurring tasks: planner, tdd-guide, code-reviewer, security-reviewer, build-error-resolver. Each has a focused role and gets invoked for the right thing rather than dumping everything at the main agent.

**Skills** — Reusable command prompts: `/plan`, `/verify`, `/simplify`, `/commit-push-pr`, `/learn`. The verify skill runs tsc → lint → tests → build in order and stops on first failure.

**Global rules** — Coding style, git workflow, security checklist, package manager preferences (bun, not npm), test coverage minimums. Claude reads these on every session and actually follows them.

## What it actually changed

I stopped re-explaining my stack. I stopped getting npm in projects that use bun. Security checks happen before commits without me prompting. The context system means I can close a session mid-task and come back three days later without losing where I was.

It's not magic — it's just configuration that enforces the workflow I actually want.

## Stack

- Claude Code hooks (bash scripts)
- CLAUDE.md (global + per-project)
- Session context (4 markdown files per project)
- Custom agents + skills

## What this is

A home lab side project. I wanted air quality monitoring in my workspace and didn't want to use a proprietary device that locks data behind an app. So I wired up a Bosch BME680 sensor to an Orange Pi 3 LTS and built the monitoring stack myself.

Tracks temperature, humidity, pressure, gas resistance, dew point, and heat index. Outputs structured JSON every 2 seconds, ready to pipe into InfluxDB, an ELK stack, or just log to disk.

## Why a dynamic baseline matters

Raw gas resistance from the BME680 is meaningless without context — the number means different things depending on the sensor's warm-up state and ambient conditions. The system runs a 120-second warmup on startup, establishes a moving-average baseline, then calculates an air quality score (0–100) relative to that baseline. The score is actually useful rather than just a raw ohm reading.

## What the output looks like

```json
{
  "temperature_c": 31.04,
  "humidity_rh": 68.55,
  "gas_baseline_ohms": 42150.5,
  "air_quality_score": 52.3,
  "dew_point_c": 24.75,
  "heat_index_c": 34.47,
  "baseline_established": true
}
```

Deployed as a Docker container with hardware access via privileged mode — whole thing starts with `docker compose up`.

## One gotcha worth noting

Position the BME680 away from the Orange Pi's CPU. I didn't, and the first week of temperature readings were 4–5°C too high. Moved it to a separate mount on the case and readings normalized.

## What this is

A maintenance automation tool I wrote because I got tired of running different update commands on different machines. One script that handles `apt`, `dnf`, `pacman`, `brew`, `npm`, `pip` — whatever's present on the system — with consistent logging and safe execution defaults.

Runs unattended. Doesn't hang waiting for user input. Doesn't overwrite config files without asking. Keeps timestamped logs so you can see what actually happened.

## Why I built it instead of using something existing

Most existing solutions are either too opinionated (full configuration management like Ansible is overkill for this) or too simple (a cron job with `apt upgrade -y` has no error handling). I wanted something in between — lightweight, readable Bash that does the job reliably and can be understood and modified without reading documentation.

## How it works

**Core + Modules architecture** — the core script handles argument parsing, logging, environment detection, and concurrency control (lockfile to prevent simultaneous runs). Each package manager is a separate module that the core calls if that manager is present on the system.

**Execution modes** — `--check-only` to see what would update without doing it, `--security-only` for security patches only, `--reboot` on Linux systems that need a restart after kernel updates.

**Scheduled via native tools** — `onboard.sh` sets up a Cron job on Linux or a Launchd agent on macOS so it runs automatically on whatever schedule you want.

## What this is

A personal project exploring what a real multi-agent system looks like in practice. The idea: instead of one general-purpose AI assistant trying to do everything, a central orchestrator (Buddy) routes work to specialized agents — each with their own instructions, toolset, and context.

29+ agents total. Things like `github-manager`, `security-scanner`, `code-reviewer`, `doc-writer`. Each is a separate configuration that Buddy loads dynamically based on what the task needs.

## Why MCP

Model Context Protocol gives agents a standardized way to expose tools. Instead of every agent needing custom integration code, they publish tools through MCP and any orchestrator can discover and call them. It's what makes the routing actually work — Buddy doesn't need to know the implementation details of each agent, just what tools it offers.

## How it works

Tasks come in to Buddy. Buddy analyzes what's needed, spawns the right agent(s), hands off context, and coordinates the results. Agents can hand off to each other — a scanner finding a vulnerability can trigger the coder agent to write a fix, which triggers the reviewer agent to validate it.

State persists through a Context Hub so long-running tasks don't lose progress between sessions.

## What I was actually trying to figure out

Whether specialized agents produce meaningfully better results than a single agent with a long system prompt. The answer is yes, but with caveats — the routing logic matters a lot, and getting agents to hand off cleanly without losing context is harder than it sounds. Still iterating on that.

## What this is

The goal was to build something I'd trust to run unattended and that I could actually learn from rather than just copy-paste from a tutorial.

## Why Swarm over Compose or Kubernetes

Kubernetes is overkill for one server. Plain Compose works but doesn't give you health-based restarts or easy service redeployment. Swarm hits the right balance — proper orchestration primitives without the operational overhead. It's been running with no manual intervention for over a year.

## How it's structured

Everything is network-segmented. External traffic comes in through Traefik on 80/443 — it handles Let's Encrypt SSL automatically and routes to the right service based on hostname. Backend services talk to each other over internal overlay networks and have no external exposure.

Secrets are managed through Infisical, not environment variables in Compose files. Took some extra setup but means I'm not looking at plaintext secrets in a git repo.

## What runs on it

- **Traefik** — edge router, SSL termination, service discovery
- **n8n** — workflow automation, connected to various APIs
- **Infisical** — secrets management for the whole stack
- **Portainer** — Swarm management UI
- **Firecrawl** — documentation crawling for AI projects
- **SearXNG** — self-hosted metasearch

## What didn't go smoothly

The first version had everything in one large Compose file. Adding a new service meant touching the global config and risking breaking something else. Refactored to a modular stack-per-service structure — each service lives in its own folder with its own Compose file and connects to the shared `proxy-net` for ingress. Much easier to maintain and deploy independently.
