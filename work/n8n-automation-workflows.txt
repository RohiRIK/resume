## What this is

The Microsoft Intune portal is fine for small fleets. At 10,000+ devices it gets slow and the search is painful. FleetWatch is a dashboard I built to solve the specific problems that come up when you're managing a large fleet day-to-day.

## Why I built it

Working with large Intune environments at Oncloud, the native portal was consistently the bottleneck during incident response — finding a specific device took too long, compliance state was stale, and triggering actions required too many clicks. I wanted a faster tool that worked the way the job actually works.

## What it does

**Fast device search** — full-text search across the entire fleet using optimized PostgreSQL indexing. Find any device, user, or serial number in under a second regardless of fleet size.

**Real-time compliance** — continuous sync with Microsoft Graph means compliance state is current, not cached. Useful when you need to know right now whether a device is actually compliant.

**Identity correlation** — links Intune device records with Entra ID user profiles automatically, so you see the full picture without switching between portals.

**Direct actions** — trigger Intune actions (sync, remote lock, restart, wipe) from the dashboard without navigating the native portal.

**Compliance trends** — shows compliance over time per policy, useful for spotting drift before it becomes a problem.

- Next.js 15 (App Router)
- PostgreSQL with GIN/B-Tree indexes
- Microsoft Graph SDK

## What this is

A passion project built because hosting friends for coffee had become genuinely chaotic. Multiple orders, specific customizations, no memory of who asked for oat milk — it was more stressful than it should be.

So I built an order management system. Guests scan a QR code, see the menu, customize their order, and submit it. I get a kitchen display showing the queue with all the details. It sounds like overkill, and it is, but it works perfectly.

## How it works

**Guest side** — a mobile-first menu where guests choose their drink, add customizations (milk type, shots, temperature), and place the order. Loads instantly, no app install needed, works as a PWA if they want it on their home screen.

**Host side (KDS)** — a kitchen display that shows the current queue. Orders appear immediately when placed, I can mark them as "preparing" or "ready", and the guest's screen updates in real time without a refresh.

Firebase Firestore real-time snapshots handle the live updates — when an order status changes on the KDS, it propagates to the guest view in under a second.

## What I actually learned from this

Firebase Firestore's real-time subscriptions are genuinely good for this use case. The tricky part was handling the edge cases — what happens when the host closes the KDS while orders are in flight, stale order states, offline behavior. None of it was hard, but it required thinking through the state machine more carefully than the happy path suggests.

Also: the first version had a bug where two guests placing orders within milliseconds of each other could get the same order number. Fixed it with a Firestore transaction. Classic concurrency problem, embarrassingly obvious in retrospect.

## Stack

- Next.js 15 (App Router)
- Bun
- Firebase Firestore (real-time)
- PWA

## What this was

A client running JumpCloud for MDM wanted to consolidate onto the Microsoft stack. Everything else was already in Microsoft — identity, email, security — but the device management gap meant correlating device state with identity required jumping between platforms.

## What I built

The main challenge with MDM migrations is the transition gap — the window where a device is unenrolled from one MDM and not yet in another. During that time you lose policy enforcement and visibility.

I wrote PowerShell scripts to handle the transition in sequence: unenroll from JumpCloud, trigger Intune enrollment, verify the device showed up in Intune before moving to the next. Ran in batches with rollback checkpoints so we could pause if something broke mid-migration.

**Windows Autopilot** — registered the Windows fleet into Autopilot for zero-touch future provisioning. Extracted hardware hashes, uploaded them, configured deployment profiles per user group.

**macOS via Apple Business Manager** — set up ABM integration with Intune. More involved than Windows Autopilot — had to work through the MDM profile trust chain and handle devices that had supervision issues from the JumpCloud side.

**Policy parity** — translated JumpCloud's Command Library into Intune Configuration Profiles and PowerShell scripts deployed via Intune. Some mapped cleanly, some needed rethinking.

## What made it hard

A subset of devices had been enrolled in JumpCloud via a method that left a persistent MDM profile the script couldn't cleanly remove. Those had to be handled manually. Also found several macOS devices enrolled by users personally rather than via ABM, which broke the supervision requirement for certain policies.

## Stack

- Microsoft Intune
- JumpCloud
- Windows Autopilot
- Apple Business Manager
- PowerShell & Microsoft Graph API

## What this was

A client at Oncloud decided to move their entire identity and collaboration stack from Google Workspace to Microsoft 365. I was the lead engineer on the migration — responsible for technical execution from discovery through cutover.

## What the work looked like

**Identity migration** — mapped thousands of Google identities into a clean Entra ID tenant, standardized naming conventions, consolidated accounts that had duplicated from acquisitions. Set up SSO from scratch with Entra ID as the new IdP.

**Data migration via AvePoint Fly** — handled mailbox and Google Drive to SharePoint/OneDrive transfer. The hard part wasn't the tool — it was the permission mapping. Google Drive's sharing model doesn't translate cleanly to SharePoint. Spent significant time building mapping rules to preserve access rather than just moving data and leaving people unable to open their files.

**Mail cutover** — ran split-delivery during the transition so mail kept flowing to both platforms while users moved over. Managed the DNS cutover window to avoid gaps.

**SSO re-mapping** — the client had 50+ SaaS applications pointed at Google as IdP. Reconfigured each for SAML/OIDC against Entra ID. Most were straightforward; a few needed custom attribute mappings.

## What made it hard

The data migration took longer than scoped. The client's Google Drive was far messier than discovery suggested — deeply nested shared drives with complex permission inheritance, orphaned files, and accounts that no longer existed but still owned content. We had to build a pre-migration cleanup process that wasn't in the original plan.

## Stack

- AvePoint Fly
- Microsoft Entra ID
- Google Workspace APIs
