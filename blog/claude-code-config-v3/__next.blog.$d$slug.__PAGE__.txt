Every IT team has some version of the same problem: the MDM says one thing, Active Directory says another, and the spreadsheet someone maintains manually is a third opinion. Nobody knows the real device count. Nobody knows which machines are stale. Asset reviews are a manual reconciliation exercise that takes days and is wrong before it's finished.

For a FinTech client, I built a pipeline that pulls from every authoritative source — AD, Intune, Azure, MDE — and produces a single unified inventory that's always current. Here's the architecture.

## The data problem

The client had devices across three categories:
- **Windows workstations**: domain-joined, Intune-enrolled, some hybrid Entra joined
- **Azure VMs**: not domain-joined, sometimes Intune-enrolled, MDE-onboarded
- **Legacy machines**: domain-joined only, no Intune, no MDE

No single system had the full picture. Intune knew about enrolled devices but didn't know about the legacy machines. AD knew about everything domain-joined but didn't know about Azure VMs or compliance state. MDE knew about onboarded endpoints but didn't map cleanly to AD objects.

The goal: a unified record per device with attributes from all sources — last seen, OS version, compliance state, MDE sensor status, assigned user, department, location.

## The architecture

Four components:

**1. Jenkins** — scheduler and orchestrator. A pipeline job runs nightly and on-demand. It calls the PowerShell scripts in sequence, handles retries, and sends a completion notification.

**2. PowerShell scripts** — data collection and normalization. One script per source, plus a merge script. Each script outputs structured JSON to a staging directory on the Jenkins agent.

**3. SharePoint List** — the inventory store and intermediate layer. PowerShell writes the merged records to a SharePoint list via the SharePoint REST API or PnP PowerShell. The list is the single source of truth that other systems can query.

**4. Power BI** — visualization layer. A Power BI report connects directly to the SharePoint list and refreshes on a schedule. This is the dashboard the IT team and management use.

## The data collection scripts

**AD collector**: Uses `Get-ADComputer` with `-Properties *` to pull all computer objects. Captures name, distinguished name, last logon timestamp, OS version, description, and organizational unit. Filters out disabled objects and objects not seen in 90 days.

```powershell
Get-ADComputer -Filter * -Properties LastLogonDate, OperatingSystem, Description |
  Where-Object { $_.LastLogonDate -gt (Get-Date).AddDays(-90) } |
  Select-Object Name, DistinguishedName, LastLogonDate, OperatingSystem, Description |
  ConvertTo-Json | Out-File "$stagingPath\ad-devices.json"
```

**Intune collector**: Uses the Microsoft Graph PowerShell SDK. Calls `Get-MgDeviceManagementManagedDevice` to get all enrolled devices with compliance state, last sync time, enrolled user, and device category.

**MDE collector**: Uses the Defender for Endpoint API (via a service principal with `Machine.Read.All`). Gets onboarding status, sensor health, last seen timestamp, and risk level for every endpoint.

**Azure VM collector**: Uses `Get-AzVM` across all subscriptions the service principal has access to. Gets VM name, resource group, location, OS, running state, and any tags with owner/department information.

## The merge script

The merge script is where the real work happens. It loads all four JSON files and joins them on device name. The join logic:

1. AD is the primary key source — every AD computer object gets a record
2. Intune records are matched to AD by device name (normalized to uppercase)
3. MDE records are matched by hostname
4. Azure VMs get their own records (no AD object)

For each merged record, the script calculates derived fields:
- `managementState`: `fully-managed` (Intune + MDE), `partially-managed` (Intune only or MDE only), `unmanaged` (AD only)
- `staleness`: days since last seen across all sources
- `complianceGap`: boolean — in Intune but not compliant

The merged output is an array of device objects written to `merged-inventory.json`.

## Writing to SharePoint

PnP PowerShell handles the SharePoint write. The script compares the merged inventory against the current SharePoint list state (pulled at the start of the run) and does a delta update: new devices are added, existing devices are updated, devices not seen in 90 days are flagged as stale.

Doing a delta update rather than a full replace matters for performance (the list had 2,000+ items) and for preserving any manual annotations added by the IT team directly in the list.

## Power BI on top

The Power BI report connects to the SharePoint list as a data source and refreshes daily. Key views:

- **Management coverage**: pie/bar showing how many devices are fully managed vs. partially managed vs. unmanaged
- **Stale devices**: list of devices not seen in 30+ days, sorted by staleness
- **Compliance gaps**: devices enrolled in Intune but not passing compliance policy
- **MDE coverage**: which devices have an active sensor vs. not

The report is embedded in a Teams tab so the IT team has it in their daily context without going to Power BI directly.

## What it replaced

Before this pipeline: a manually-maintained Excel spreadsheet updated every quarter. It was always wrong. MDE coverage reports were run manually when someone asked. Compliance gaps were invisible until a CA policy blocked a user.

After: a nightly automated run, a SharePoint list the IT team trusts, and a Power BI dashboard the security team uses in their weekly review.

The pipeline has run without manual intervention for over a year. When a new device shows up as unmanaged, the report flags it. When a compliance gap appears, the security team sees it the next morning.

---

The architecture is not complicated. Jenkins, PowerShell, SharePoint, Power BI are tools every Microsoft-focused IT team already has. The value is in putting them together deliberately and removing the manual reconciliation step entirely.

Most identity platforms are configured once and then drift. The IdP has the org chart from six months ago. The HR system has been updated 50 times since then — new hires, promotions, reorgs, contractor conversions — and none of it flowed downstream.

The result: Active Directory has the wrong department for half the engineering team, Entra ID shows reporting lines from two restructures ago, and the extension attributes that should drive Conditional Access logic are either empty or stale. The HR system has clean data. The IdP does not.

For an OT Security client, I built a sync between HiBob (their HRIS) and Entra ID / on-premises AD that runs on a schedule and keeps 18 attributes current. Here's how it works.

## What HiBob knows that Entra doesn't

HiBob is the HR system of record. It owns:
- Department and sub-department
- Job title and job level
- Manager (reporting line)
- Employment type (full-time, part-time, contractor)
- Start date and tenure
- Site/office location
- Cost center
- Custom fields: clearance level, team function, contractor company

Entra ID has fields that map to most of these — `department`, `jobTitle`, `manager`, `employeeType`, `extensionAttribute1–15` — but they're only as current as the last time someone manually updated them. At most clients, that's "not recently."

## The architecture

Three components:

**HiBob API** — the source. HiBob has a REST API with token-based auth. The key endpoints are `/v1/people` (list all employees) and `/v1/people/{id}` (get individual profile). The API returns rich profile data including all custom fields.

**Azure Automation Runbook** — the sync engine. A PowerShell runbook runs on a schedule (nightly + on-demand for urgent updates). It calls the HiBob API, normalizes the data, and writes changes to Entra ID via Microsoft Graph and to on-premises AD via the AD module.

**Entra ID + on-premises AD** — the targets. The runbook writes to both. For most attributes, Entra ID is updated via Graph API and the change propagates to AD via Entra Connect sync. For attributes that only exist on-prem (like certain extension attributes not synced to Entra), the runbook writes directly to AD.

## The sync logic

The runbook follows a compare-then-write pattern to avoid unnecessary writes:

1. Pull all active employee records from HiBob API
2. For each employee, look up the corresponding Entra ID user (matched by email or employee ID stored in `employeeId` attribute)
3. Compare the HiBob values against current Entra ID attribute values
4. If any value differs, write the update via Graph API
5. Log every change with before/after values to a Log Analytics workspace

This means the runbook only writes when there's an actual difference — not on every run, not for every user. This keeps the audit log clean (changes are meaningful) and avoids triggering unnecessary sync cycles in Entra Connect.

## What gets synced

The 18 attributes synced in this implementation:

| HiBob Field | Entra ID Attribute |
|---|---|
| Department | `department` |
| Job Title | `jobTitle` |
| Manager | `manager` (reference) |
| Employment Type | `employeeType` |
| Office Location | `officeLocation` |
| Cost Center | `extension_attribute_1` |
| Contractor Flag | `extension_attribute_2` |
| Clearance Level | `extension_attribute_3` |
| Team Function | `extension_attribute_4` |
| Contractor Company | `extension_attribute_5` |
| Start Date | `employeeHireDate` |
| Display Name | `displayName` |
| Preferred Name | (custom) |
| Site | `usageLocation` |
| Employee ID | `employeeId` |
| Sub-Department | `extension_attribute_6` |
| Job Level | `extension_attribute_7` |
| Profile Picture | (via separate process) |

The manager sync is the most complex. HiBob stores the manager as an employee ID reference. The runbook resolves this to an Entra ID object ID and sets the `manager` attribute as a directory reference — not just a display name string.

## Profile picture sync

Profile pictures are handled separately. HiBob stores profile photos accessible via the API. The runbook downloads the photo, resizes it to the Microsoft 365 preferred size (648×648), and uploads it via the Graph API (`/users/{id}/photo/$value`). This keeps profile pictures in M365, Teams, and Outlook current — which matters more than it sounds for a distributed team.

## Contractor detection and Conditional Access

One of the higher-value outcomes of this sync is accurate contractor flagging. HiBob has an employment type field that distinguishes full-time employees from contractors. The runbook sets `extension_attribute_2` to `contractor` or `employee` accordingly.

This attribute drives Conditional Access policies. Contractors get a stricter policy set: phishing-resistant MFA required for all apps, no persistent browser sessions, stricter sign-in frequency. Full-time employees get the standard policy set with SSO and longer session lifetimes.

Without the sync, this distinction was manually maintained and frequently wrong — contractors who converted to FTE still had contractor-tier policies, and new contractors started with FTE-tier access until someone noticed.

## What breaks

**Employee ID mismatches.** If HiBob and Entra ID have inconsistent employee IDs (which happens during migrations, acquisitions, or manual ID assignment), the sync can't match records. We built a reconciliation report that flags unmatched HiBob employees weekly so the IT team can fix the mapping manually.

**Manager chains with circular references.** A misconfigured org chart in HiBob can create circular manager references that break the Graph API call. The runbook catches these and logs them rather than failing.

**API rate limits.** HiBob's API has rate limits. For orgs with 500+ employees, a naive implementation that calls `/v1/people/{id}` for every employee will hit them. Use `/v1/people` with field expansion to pull all employees in one call.

## 1.5 years of runtime

This runbook has been running for over 18 months at this client. It has processed thousands of attribute updates — reorgs, promotions, contractor conversions, new hires, terminations (though terminations have their own dedicated workflow). The Entra ID directory now reflects HiBob accurately. Conditional Access policies that depend on employment type work correctly. The IT team no longer gets support tickets about wrong department values.

---

The pattern generalises to any HRIS. Workday, BambooHR, Personio — if it has an API and you have Entra ID, you can build this sync. The core principle is the same: the HR system is the source of truth for people data. The IdP should be downstream of it, not a separate silo that drifts.
